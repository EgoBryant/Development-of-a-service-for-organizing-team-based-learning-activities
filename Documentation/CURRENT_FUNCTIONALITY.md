# Текущий функционал backend-проекта

Это описание актуального состояния кода после доработок MVP-ядра. Файл предназначен для ручной проверки через Swagger, Postman или любой HTTP-клиент.

## 1. Что уже работает

### Авторизация
- Регистрация пользователя через `POST /api/auth/register`.
- Логин через `POST /api/auth/login`.
- Получение профиля текущего пользователя через `GET /api/auth/me`.
- Выдача JWT после успешного логина или регистрации.

### Пользователь
- У пользователя есть `Id`, `UserName`, `Email`, `Role`.
- Пользователь может быть привязан только к одной команде.
- В профиле возвращаются данные о текущей команде, если она есть.

### Команды
- Получение списка всех команд через `GET /api/teams`.
- Получение конкретной команды через `GET /api/teams/{id}`.
- Создание команды через `POST /api/teams`.
- Вступление в команду по invite code через `POST /api/teams/join`.
- Получение своей команды через `GET /api/teams/me`.

### Админские операции
- Обновление score команды через `PATCH /api/teams/{id}/score`.
- Этот эндпоинт доступен только роли `Admin`.

### Инфраструктура
- Есть `Swagger` в режиме разработки.
- Есть `healthcheck` по `GET /health`.
- Есть `ProblemDetails` и базовая обработка ошибок.
- Есть `CORS` политика для внешнего фронтенда.

## 2. Что хранится в БД

### User
- `Id`
- `UserName`
- `Email`
- `PasswordHash`
- `Role`
- `TeamId`
- `CreatedAtUtc`

### Team
- `Id`
- `Name`
- `InviteCode`
- `Score`
- `CaptainUserId`
- `CreatedAtUtc`

### Связи
- Один пользователь может принадлежать только одной команде.
- У команды есть капитан.
- У команды есть список участников.
- `InviteCode` уникален.

## 3. Дефолтные тестовые данные

После первого запуска и миграции создаются:
- `admin@teamexam.local` / `Admin123!`
- `student@teamexam.local` / `Student123!`

Также создаются две команды:
- `Alpha`
- `Beta`

У них уже есть `InviteCode`, который можно увидеть через `GET /api/teams`.

## 4. Как тестировать вручную

### Сценарий 1. Регистрация
1. Отправить `POST /api/auth/register`.
2. Передать `userName`, `email`, `password`.
3. Проверить, что в ответе есть `token` и данные профиля.

### Сценарий 2. Логин
1. Отправить `POST /api/auth/login`.
2. Передать email и пароль.
3. Проверить JWT и поля профиля в ответе.

### Сценарий 3. Просмотр профиля
1. Взять JWT из login/register.
2. Передать `Authorization: Bearer <token>`.
3. Открыть `GET /api/auth/me`.
4. Проверить, что возвращаются данные пользователя и его команда.

### Сценарий 4. Создание команды
1. Авторизоваться под пользователем без команды.
2. Отправить `POST /api/teams`.
3. Передать только `name`.
4. Проверить, что команда создана, сгенерирован `inviteCode`, пользователь стал капитаном.

### Сценарий 5. Вступление в команду
1. Авторизоваться под пользователем без команды.
2. Отправить `POST /api/teams/join`.
3. Передать `inviteCode` существующей команды.
4. Проверить, что пользователь привязался к команде.

### Сценарий 6. Просмотр своей команды
1. Авторизоваться пользователем, который уже состоит в команде.
2. Открыть `GET /api/teams/me`.
3. Проверить состав, score, капитана и invite code.

### Сценарий 7. Админское обновление score
1. Авторизоваться под `admin@teamexam.local`.
2. Отправить `PATCH /api/teams/{id}/score`.
3. Передать новое значение `score`.
4. Проверить, что score команды обновился.

## 5. Ограничения текущей версии

- Нет фронтенда.
- Нет импорта оценок из учебного портала.
- Нет истории изменений рейтинга.
- Нет ачивок, ленты, челленджей, биржи знаний и механики спасения.
- Нет полноценного разделения прав для капитана команды, кроме базовой логики создания и админского изменения score.

## 6. Что считать готовым MVP на этом этапе

Минимально уже можно проверить такой путь:
1. Пользователь регистрируется или логинится.
2. Получает JWT.
3. Создает команду или вступает в существующую.
4. Смотрит свою команду.
5. Видит список всех команд и их рейтинг.
6. Админ меняет score команды.

Если нужно, следующим шагом можно добавить отдельный файл с готовыми `curl`-запросами и примерами JSON для каждого эндпоинта.
